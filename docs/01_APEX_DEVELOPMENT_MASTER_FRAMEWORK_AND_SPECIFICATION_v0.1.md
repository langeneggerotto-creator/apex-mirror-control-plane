# 1. Vision Kernel

APEX Development is the governed build layer that converts vision, research, dimensional understanding, learning, analytics experiences and testing into traceable, evidence-bearing artifacts.

```text
APEX Development exists to transform important ideas into tested, understandable and useful systems without allowing complexity, automation or aspiration to outrun truth, evidence or human control.
```

## 1.1 Governing Laws

| Law | Operational Meaning |
|---|---|
| Truth Before Aesthetics | Every module distinguishes what exists, what is designed, what is tested and what is future vision. |
| Evidence Before Promotion | No module moves to validated/deployed status from architecture or presentation quality alone. |
| Human Control Supremacy | Otto retains authority to stay, switch, hold, stop, approve or redirect development. |
| SGNUM+ Output Standard | Outputs are standardized, generalized, normalized, universalized and minimized without losing needed meaning. |
| C∞ Convergence Option | Every bounded build may iterate toward proxy completion gap `MSE ≈ 0.0001` where a meaningful metric exists; this is build-completeness convergence, not real-world proof. |
| Final Direction Stack | Every meaningful module output ends with three ranked implementation directions and one sandboxed innovation direction. |
| Public-Safe GitHub Boundary | Public repositories receive only public-safe artifacts; sensitive evidence requires controlled storage. |

# 3. Unified Development Lifecycle

Every APEX development workstream follows the same development lifecycle, while producing domain-specific outputs.

```text
Observer Signal
→ Vision Kernel
→ Current-State Inventory
→ Requirements & Constraints
→ Design / Model / Architecture
→ Build Artifact
→ Evidence & Tests
→ Promotion Decision
→ Benefit Measurement
→ Learning / Patch / Recalculation
```

## 3.1 Lifecycle Gates

| Gate | Required Question | Minimum Output | Allowed Status After Gate |
|---:|---|---|---|
| `G0 — Origin` | What exists and why does this module matter? | Charter and truth boundary. | `SEED` |
| `G1 — Definition` | What exactly will it do and not do? | Requirements and scope. | `DEFINED` |
| `G2 — Architecture` | How will it function and integrate? | Technical/functional specification. | `DESIGNED_NOT_PROVEN` |
| `G3 — Prototype` | Can a real artifact be produced? | Runnable or reviewable artifact. | `PROTOTYPE` |
| `G4 — Validation` | Did it pass declared tests? | Test output and evidence receipt. | `TESTED` |
| `G5 — Pilot` | Does it work in a bounded real use case? | Pilot evidence and measured results. | `PILOT_VALIDATED` |
| `G6 — Controlled Use` | Are controls, ownership and rollback sufficient? | Approved controls and operating record. | `GOVERNED_USE` |
| `G7 — Evolution` | Did results justify improvement or expansion? | Benefit review and revised direction stack. | `ITERATING` or `PROMOTED` |

## 3.2 Truth Status Vocabulary

| Status | Use When |
|---|---|
| `VERIFIED` | Evidence directly establishes the claim. |
| `INFERRED` | Reasonable conclusion derived from identified evidence, not directly confirmed. |
| `DESIGNED_NOT_PROVEN` | Specification or architecture exists without operational validation. |
| `PROTOTYPE` | Artifact is runnable/reviewable but not validated for operational use. |
| `TESTED` | Named tests executed and passed for declared scope only. |
| `BLOCKED` | Evidence, ownership, input, permission or safety requirement prevents progression. |
| `ASPIRATIONAL` | Future-state possibility, not active capability. |

## 7.2 Dimensional Ontology Use Boundary

The dimension ladder is a **modeling-depth and maturity framework**. It is not proof that software, learning or organizations occupy physical dimensions beyond accepted scientific usage.

## 7.3 Core Modeling Ladder

| Dimension | System Question | Development Meaning |
|---:|---|---|
| `-1D` | Why is there something rather than nothing? | Existence / origin / potential. |
| `0D` | What is observed? | Raw signal, truth baseline and captured evidence. |
| `1D` | What is it? | Identity and entity definition. |
| `2D` | How is it built? | Components and structure. |
| `3D` | How does it appear? | Presentation and interface. |
| `4D` | Where is it? | Space, location, placement and context position. |
| `5D` | When/how does it change? | Time, sequence and history. |
| `6D` | What condition is it in? | State and health/status. |
| `7D` | What surrounds it? | Context and environment. |
| `8D` | What does it do? | Behavior and action. |
| `9D` | What limits it? | Constraints and boundaries. |
| `10D` | What connects it? | Relationships, networks and dependencies. |
| `11D` | What happens next? | Prediction and forecast under evidence. |
| `250D+` | How do signals combine? | Multi-modal integration aspiration / later modeled capability. |
| `750D+` | How does it learn and adapt? | Adaptive intelligence aspiration / later modeled capability. |
| `2000D+` | Why does it matter? | Purpose/wisdom impact model aspiration. |
| `∞D` | How does everything unify under governance? | Unified reality/system-field direction only. |

# 12. Shared Non-Functional Requirements

| Requirement Area | Standard |
|---|---|
| Truth | No unsupported maturity, deployment, operational or performance claim. |
| Explainability | A non-specialist can understand what was built and why. |
| Evidence | Meaningful claims link to artifacts/tests/records. |
| Security | No credentials or protected content in public repositories. |
| Privacy | Personal and enterprise-sensitive information excluded or controlled. |
| Accessibility | UI artifacts should target keyboard operation, readable contrast and understandable labels. |
| Reversibility | Material build changes have rollback or correction path. |
| Interoperability | Schemas and output contracts are explicitly versioned. |
| Automation | Automation may advance approved safe work; it must not bypass evidence or authority gates. |
| Efficiency | Reduce avoidable complexity, duplication and token/work waste while preserving critical context. |

# 13. Maturity and Measurement Model

## 13.1 Portfolio Measures

| Metric | Purpose |
|---|---|
| Artifact Completion | What required outputs exist. |
| Evidence Coverage | Percentage of meaningful claims tied to evidence records. |
| Test Pass Rate | Declared acceptance gates passed for bounded scope. |
| Blocker Severity | Outstanding issues that prevent promotion. |
| Human Approval State | Whether consequential decisions are approved. |
| Benefit Score | Expected or measured payoff, labelled by evidence level. |
| Rework / Waste Rate | Avoidable iteration or duplicate work identified. |
| Convergence Gap | Remaining build-completeness gap for bounded scope. |

## 13.2 C∞ Convergence Rule

For a bounded build with measurable completion criteria, the Iteration Automater may later calculate a proxy completion loss:

```text
C∞ Loss = weighted gap between required, completed, validated and governable outputs.
```

Target option:

```text
Iterate until proxy build-completeness MSE ≤ approximately 0.0001,
or until hold/stop/no-significant-gain/safety/evidence rule triggers.
```

This is not proof of real-world quality, business value, learning effect, universal portability or production readiness.
