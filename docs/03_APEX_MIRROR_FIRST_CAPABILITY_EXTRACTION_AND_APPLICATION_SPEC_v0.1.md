# 3. Most Valuable Principles to Adopt Immediately

## 3.1 Mirror First, Not Automation First

The Mirror must become the first visible surface of the portfolio. The Iteration Automater may later progress bounded tasks beneath it, but it must not become the invisible center of authority.

```text
Correct Architecture
Human Authority / Value Alignment
          ↓
APEX Mirror — visible truth, evidence, payoff, controls
          ↓
Iteration Automater OS — bounded execution only
          ↓
Module builds and proof artifacts
```

**Module Application:** the Control Plane receives state/evidence from every module and only emits approved work orders to the automater.

## 3.2 Truth Before Synthetic Dashboard Confidence

The visual anchors use scores such as integration percentages, evidence packages, cycles completed, health percentages and “active/live” markers. These are useful **display archetypes**, but not evidence of existing performance.

### Implementation Rule

Every metric card must support one of these data states:

| Display State | Meaning | Permitted Display |
|---|---|---|
| `MEASURED` | Real recorded metric with source and date. | Value + evidence source + freshness. |
| `VERIFIED_STATUS` | Artifact/test/commit establishes binary or categorical state. | Status badge + evidence link. |
| `DEMO_DATA` | Illustrative/sample value for UI review. | Value only with persistent demo label. |
| `DESIGNED_NOT_CONNECTED` | UI exists but no metric source is connected. | `Not measured / source pending`. |
| `BLOCKED` | Required evidence/input unavailable. | Block reason + resolution action. |

## 3.3 Human Authority Must Be Visible, Not Implied

The most valuable safety improvement from the imagery is a persistent authority layer above automated progress.

### Required Decisions

| Control | User Meaning | Automation Boundary |
|---|---|---|
| `STAY` | Continue current ranked activity. | Automater may progress approved bounded task. |
| `SWITCH` | Choose another direction for payoff or any user reason. | Old path is paused; new path requires declared scope and gate. |
| `HOLD` | Stop progress pending proof/input/approval. | No automated execution. |
| `STOP` | End or archive an unproductive/unsafe path. | Preserve ledger and rollback record. |
| `INNOVATE` | Explore alternative in sandbox. | May not overwrite active path or promotion status. |

## 3.4 Every Module Needs a Live-but-Truthful Status Card

Each visible workstream from the development canvas must eventually appear as a standardized card in the Mirror.

| Standard Card Field | Required Meaning |
|---|---|
| Module Name / Symbol | Clear identity. |
| Purpose Kernel | Why it exists. |
| Current Status | Verified, designed, prototype, tested, blocked or aspirational. |
| Current Artifact | Latest material artifact or `none yet`. |
| Evidence State | Evidence receipt / missing / pending. |
| Benefit Hypothesis | Expected value, labelled inferred until measured. |
| Current Risk / Blocker | Main constraint or gap. |
| Next Proof Gate | What must be done next to advance. |
| User Control | Stay, switch, hold, stop, innovate. |

# 6. Evidence-Safe Metrics and KPI Model

## 6.1 Metrics the Mirror May Display Once Data Exists

| KPI | Meaning | Minimum Source Needed | Safe Interim Display |
|---|---|---|---|
| Evidence Coverage | Portion of meaningful feature claims with evidence receipts. | Evidence Ledger records. | `Not measured — schema in progress`. |
| Module Connectivity | Recorded and validated module interface connections. | Integration contracts + test results. | `Designed connections only`. |
| Build Completion | Required artifacts created for a bounded package. | Artifact registry. | Count of committed design artifacts only. |
| Validation Pass Rate | Tests passed for declared feature scope. | Test Workbench evidence. | `No executed tests connected`. |
| Human Approval State | Actions explicitly approved/held/stopped. | Decision records. | `Approval framework designed`. |
| Benefit Realization | Verified outcomes after use. | Measured outcome record. | `Benefit hypothesis only`. |
| Convergence Gap | Remaining scoped build-completeness gap. | C∞ convergence record. | `Not calculated`. |
| Risk / Blocker State | Critical gaps preventing promotion. | Blocker ledger. | Visible known gaps from design stage. |

## 6.2 Forbidden Metric Behavior

The Mirror must not:

- display percentage scores without an origin record or persistent `DEMO` label;
- display `ACTIVE`, `LIVE`, `INTEGRATED`, `DEPLOYED`, `VALIDATED` or `AUTONOMOUS` unless evidence supports the exact scope;
- treat a concept visual, prompt, specification, screenshot or repository existence as proof of a functioning subsystem;
- combine aspirational human/digital-twin outcomes with present-state performance measures.

# 7. Governance and Guardrail Gate Set

The strongest usable guardrails extracted from the images become mandatory control-plane gates:

| Guardrail | Control-Plane Implementation | Blocks Promotion When |
|---|---|---|
| Truth | Claim status, evidence reference and unknowns visible. | Claim is unsupported or falsely promoted. |
| Safety | Risk assessment and safe-stop requirements. | Consequential risk lacks mitigation. |
| Privacy | Public/private data boundary. | Confidential or sensitive information is proposed for public repository/display. |
| Transparency | Human-readable decision and change history. | Decision source or logic cannot be inspected. |
| Consent | Appropriate human approval for impacts. | Approval is required but absent. |
| Accountability | Owner and rollback or correction path. | No accountable owner/control path exists. |
| Human Dignity | Human benefit and non-harm check. | Outcome reduces agency, dignity or safe opportunity without justification. |
